# PRD: 轻量级 Claude Code 多模型代理

## Problem Statement

用户需要通过 Claude Code CLI 使用多种上游 LLM 提供商（DeepSeek、智谱 GLM、OpenCode Go），但 Claude Code 原生只支持 Anthropic API。现有的开源代理方案 `free-claude-code` 功能过于臃肿（包含消息平台、Admin UI、Codex 支持、15+ Provider），且依赖 Python 3.14 运行时，部署和维护成本高。

用户需要一个**轻量级、单文件可运行**的代理，只做核心的协议转换和模型路由，支持思考模式的可配置开关与级别控制，以及 1M 上下文的透传兼容。

## Solution

构建一个 TypeScript 精简代理服务，对外暴露 Anthropic Messages API 兼容端点（`POST /v1/messages`），根据模型路由将请求转发到对应的上游 Provider：

- **DeepSeek**：直接透传原生的 Anthropic Messages 格式（DeepSeek 支持 `/anthropic/messages` 端点），零协议转换
- **智谱 GLM**：将 Anthropic Messages 转换为 OpenAI Chat Completions 格式，SSE 流式回传时反向转换
- **OpenCode Go**：同智谱，转换为 OpenAI Chat Completions 格式

思考模式通过环境变量统一配置，支持全局开关 + 四级思考深度（off/low/high/max），代理层自动翻译为各 Provider 的原生参数格式。1M 上下文不做任何截断，完全透传客户端的 max_tokens 和 system prompt。

## User Stories

1. As a Claude Code 用户，我希望设置 `ANTHROPIC_BASE_URL` 指向本地代理后，所有请求自动路由到 DeepSeek，无需修改 Claude Code 的任何使用习惯
2. As a Claude Code 用户，我希望通过环境变量切换模型（如 `MODEL=zhipu/glm-5.2`），代理自动选择对应 Provider 并完成协议转换
3. As a 开发者，我希望通过 `ENABLE_THINKING=false` 全局关闭思考模式，减少 token 消耗
4. As a 开发者，我希望通过 `THINKING_LEVEL=high` 控制思考深度，在不同场景下平衡推理质量和成本
5. As a 重度用户，我希望发送超过 200K tokens 的上下文时，代理不做任何截断，完整透传到上游
6. As a 运维人员，我希望通过 `GET /health` 检查代理健康状态
7. As a 运维人员，我希望通过 `GET /v1/models` 查看当前可用的模型列表
8. As a 开发者，我希望代理在启动时验证 Provider API Key 是否配置且有效
9. As a 开发者，我希望在不支持的模型名被请求时，代理返回清晰的错误信息
10. As a 调试用户，我希望通过 `LOG_LEVEL=debug` 查看请求和响应的元数据日志（不含敏感内容）

## Implementation Decisions

### 技术选型

- **语言/运行时**: TypeScript，Node.js 18+（使用内置 `fetch`，无需额外 HTTP 客户端依赖）
- **依赖策略**: 最小化依赖。仅使用 `typescript` + `tsx`（运行） + `zod`（配置校验）。不使用 Express/Fastify 等框架，直接用 `node:http` 构建 HTTP 服务
- **包管理**: pnpm（优先）或 npm

### 模块架构

项目由以下模块组成：

1. **config** — 环境变量解析与校验，思考级别枚举定义，Provider 元数据（端点 URL、模型列表）
2. **server** — HTTP 服务生命周期，路由注册，启动/优雅关闭
3. **model-router** — 模型名 → Provider 解析（支持 `provider/model` 格式和别名映射）
4. **providers/base** — Provider 抽象接口：`streamResponse(request, thinkingEnabled) → AsyncIterator<SSEEvent>`
5. **providers/deepseek** — DeepSeek Provider：直接透传 Anthropic 请求到 `/anthropic/messages`，逐块转发 SSE
6. **providers/zhipu** — 智谱 Provider：Anthropic→OpenAI 转换 → 调用 `/chat/completions` → OpenAI SSE→Anthropic SSE
7. **providers/opencode-go** — OpenCode Go Provider：同智谱，目标端点为 `https://opencode.ai/zen/go/v1/chat/completions`
8. **conversion/anthropic-to-openai** — Anthropic Messages 消息格式转 OpenAI Chat 消息格式（含 system prompt、tools、tool_choice、thinking→reasoning_effort 映射）
9. **conversion/openai-sse-to-anthropic** — OpenAI Chat Completions SSE 事件转 Anthropic Messages SSE 事件（含 reasoning_content→thinking delta、tool_calls→tool_use block）
10. **sse** — Anthropic SSE 事件格式化工具（message_start、content_block_start/delta/stop、message_delta、message_stop）
11. **thinking** — 思考级别→Provider 原生参数映射表

### Provider 端点与协议

| Provider | 上游协议 | 端点 | 是否需要协议转换 |
|----------|---------|------|:---:|
| DeepSeek | Anthropic Messages (原生) | `https://api.deepseek.com/anthropic/messages` | 否（透传） |
| 智谱 GLM | OpenAI Chat Completions | `https://open.bigmodel.cn/api/paas/v4/chat/completions` | 是 |
| OpenCode Go | OpenAI Chat Completions | `https://opencode.ai/zen/go/v1/chat/completions` | 是 |

### 思考模式参数映射表

统一的四级配置，代理层按 Provider 自动翻译：

| 用户级别 | DeepSeek (Anthropic 透传) | 智谱 GLM | OpenCode Go |
|----------|--------------------------|----------|-------------|
| `off` | `thinking: { type: "disabled" }` | `thinking: { type: "disabled" }` | 不发送 thinking 参数 |
| `low` | `budget_tokens: 4000` | `reasoning_effort: "high"`（low 被映射为 high） | `reasoning_effort: "minimal"` |
| `high` | `budget_tokens: 16000` | `reasoning_effort: "high"` | `reasoning_effort: "high"` |
| `max` | `budget_tokens: 32000` | `reasoning_effort: "max"` | `reasoning_effort: "xhigh"` |

环境变量：

```env
# 全局
ENABLE_THINKING=true          # 总开关
THINKING_LEVEL=high            # off | low | high | max

# 按 Provider 覆盖（可选）
DEEPSEEK_THINKING_LEVEL=max
ZHIPU_THINKING_LEVEL=high
OPENCODE_GO_THINKING_LEVEL=low
```

### 模型路由规则

1. 如果请求的 `model` 字段包含 `/`（如 `deepseek/deepseek-v4-pro`），解析为 `provider_id/model_id`，直接路由
2. 否则 fallback 到默认 Provider + 默认模型（由 `DEFAULT_MODEL` 环境变量设定）
3. 支持简写别名：`sonnet` → 默认模型；`opus` / `haiku` 同理

### Temperature 配置

各 Provider 对 temperature 的支持存在差异，代理层做统一处理：

| Provider | 默认值 | 有效范围 | 特殊约束 |
|----------|--------|---------|---------|
| DeepSeek | 1.0 | 0.0 – 2.0（推荐 0.0 – 1.5） | 无特殊限制 |
| 智谱 GLM | 0.6 | **(0, 1)** 开区间 | **不能为 0**（与 OpenAI 不同），需 clamp 到 ≥0.01 |
| OpenCode Go | 1.0 | 0.0 – 2.0（标准 OpenAI） | 取决于上游实际模型 |

环境变量：

```env
# 全局默认（client 请求中传了 temperature 则以 client 为准）
TEMPERATURE=1.0

# 按 Provider 覆盖默认值（仅当 client 未显式传 temperature 时生效）
DEEPSEEK_TEMPERATURE=1.0
ZHIPU_TEMPERATURE=0.6
OPENCODE_GO_TEMPERATURE=1.0
```

**优先级规则：**
1. Client 请求中的 `temperature` 字段 → 直接透传（最高优先级）
2. 若 client 未传 → 使用 Provider 级环境变量（`{PROVIDER}_TEMPERATURE`）
3. 若 Provider 级也未设置 → 使用全局 `TEMPERATURE`
4. 全部未设置 → 使用 Provider 默认值

**智谱特殊处理：** 最终解析出的 temperature 值如果 ≤ 0，自动 clamp 到 `0.01`（因为智谱要求范围 (0, 1) 开区间，不接受 0）。

**top_p 透传：** `top_p` 参数不做任何处理，client 传了就透传，不传则不发。

### 1M 上下文透传策略

- 不做任何内容截断或 token 限制
- `max_tokens` 字段原样透传到上游请求
- 上游返回的 `output_tokens` 原样填入 SSE `message_delta`
- 不为 1M 上下文场景做特殊处理——依赖上游 Provider 自身的上下文窗口

### Anthropic→OpenAI 消息转换规则

参考 `free-claude-code` 的 `AnthropicToOpenAIConverter`，核心转换逻辑：

- `system` (string 或 content block list) → messages 数组中的 `role: "system"` 消息
- `user` 消息的 `text` block → `role: "user"` + `content: string`
- `user` 消息的 `tool_result` block → `role: "tool"` + `tool_call_id` + `content`
- `assistant` 消息的 `text` block → `role: "assistant"` + `content: string`
- `assistant` 消息的 `thinking` block → `<think>` 标签包裹（think_tags 模式）
- `assistant` 消息的 `tool_use` block → `role: "assistant"` + `tool_calls[]`
- `tools` 数组 → OpenAI `tools` 数组（`type: "function"`）
- `tool_choice` → OpenAI `tool_choice`
- `stop_sequences` → OpenAI `stop`

### OpenAI SSE→Anthropic SSE 转换规则

- `delta.reasoning_content` → Anthropic `thinking_delta` 事件
- `delta.content` → `<think>` 标签解析后：
  - 标签内文本 → `thinking_delta`
  - 标签外文本 → `text_delta`
- `delta.tool_calls` → Anthropic `tool_use` block（content_block_start + input_json_delta + content_block_stop）
- `finish_reason` → Anthropic `stop_reason`：
  - `stop` → `end_turn`
  - `length` → `max_tokens`
  - `tool_calls` → `tool_use`

### HTTP API 契约

**POST /v1/messages** — 核心端点
- 请求：Anthropic Messages 格式 JSON
- 响应：`text/event-stream`（Anthropic SSE 格式）
- 认证：`x-api-key` header 或 `Authorization: Bearer` header（可选，由 `ANTHROPIC_AUTH_TOKEN` 控制）

**GET /v1/models** — 模型列表
- 响应：`{ data: [{ id: string, ... }] }`

**GET /health** — 健康检查
- 响应：`{ status: "ok" }`

### 已知风险

- **OpenCode Go thinking 穿透 bug**：GitHub issue #27361 报告 model 级别 options（reasoning/thinking）在 headless 模式下可能被静默丢弃。这意味着通过 OpenCode Go 启用思考模式可能不稳定，需在实际环境中验证
- **智谱 reasoning_effort 模型限定**：`reasoning_effort` 仅在 GLM-5.2 上生效，其他模型仅支持 `thinking.type` 开关
- **智谱 reasoning_content 回传要求**：多轮对话中，包含 `reasoning_content` 的 assistant 消息必须原样放回 `messages[]`，否则请求被拒

## Testing Decisions

### 测试策略

采用 **HTTP 端点契约测试** 作为最高层测试接缝（seam），覆盖完整请求→响应链路：

1. **Seam 1: HTTP 端点契约测试**（最高层，推荐主覆盖）
   - 启动本地 server + mock 上游 HTTP
   - 发送 Anthropic 格式 `POST /v1/messages`
   - 验证返回的 SSE 事件序列完整性和格式正确性
   - 一个测试覆盖路由 + 转换 + 流式输出全链路

2. **Seam 2: 消息转换单元测试**
   - 输入：Anthropic Messages JSON
   - 输出：OpenAI Chat Completions JSON
   - 覆盖：纯文本、tool_use + tool_result 往返、thinking block 回放

3. **Seam 3: SSE 转换单元测试**
   - 输入：OpenAI SSE 事件序列
   - 输出：Anthropic SSE 事件序列
   - 覆盖：reasoning_content delta、content delta（含 `<think>` 标签）、tool_calls delta

4. **Seam 4: 思考配置单元测试**
   - 输入：`THINKING_LEVEL` + Provider 类型
   - 输出：Provider 原生参数（type, budget_tokens, reasoning_effort）
   - 覆盖：四级映射 + 边界值

### 测试原则

- 只测外部行为，不测内部实现
- 使用 `node:test`（内置）或 `vitest`
- Mock 上游 HTTP 而非真实调用

## Out of Scope

- 消息平台集成（Discord/Telegram bots）
- Admin UI 或 Web 管理面板
- OpenAI Responses API（Codex 支持）
- CLI launchers（fcc-claude 等）
- 本地模型支持（Ollama、LM Studio、Llama.cpp）
- Voice 转录
- Web search / web fetch 本地工具
- 流式恢复 / retry / holdback 机制（v1 不做，后续按需）
- 多用户 / 租户隔离
- Rate limiting（依赖上游 Provider 自身限制）
- 模型发现 / 动态模型列表刷新（使用静态配置）
- 除 DeepSeek、智谱、OpenCode Go 以外的 Provider
